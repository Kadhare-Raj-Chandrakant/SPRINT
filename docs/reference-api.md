# Reference: API & Endpoints

`glint` exposes a small HTTP surface: a GitHub webhook, a liveness health
check, a read-only dashboard, and signed one-click action links. All paths are
relative to the configured `server.port` (default `8080`).

> Auth model: the webhook is authenticated by GitHub's HMAC signature; the
> dashboard is read-only; action endpoints require a valid signed token. There
> is no user login.

## `POST /webhook/github`

GitHub's webhook target. Receives `pull_request` events.

**Headers (from GitHub):**

| Header | Purpose |
|---|---|
| `X-Hub-Signature-256` | `sha256=` + HMAC of the raw body with `GITHUB_WEBHOOK_SECRET`. |
| `X-GitHub-Event` | Event name (must be `pull_request`). |
| `X-GitHub-Delivery` | Unique delivery id (used as `traceId` in logs). |

**Behavior:**

1. `WebhookSignatureVerifier` recomputes the HMAC and rejects with `401`/`403`
   on mismatch.
2. If `event != pull_request`, returns `200 OK` (ignored) without processing.
3. If `action ∈ {opened, synchronize, reopened}`, enqueues
   `ReviewOrchestrator.processPullRequest` asynchronously and returns `202
   ACCEPTED` immediately. Other actions return `200 OK` (ignored).
4. Duplicate deliveries for the same `owner/repo/pr/commitSha` are skipped.

**Responses:**

| Status | Meaning |
|---|---|
| `202 Accepted` | PR review enqueued. |
| `200 OK` | Event received but not actionable (ignored). |
| `401` / `403` | Signature verification failed. |

The handler reads the raw body (not the parsed JSON) so the HMAC can be verified
against the exact bytes GitHub sent.

## `GET /webhook/health`

Liveness probe. Returns `200 OK` with body `OK`. Used by load balancers; it does
**not** reflect GitHub/LLM/mail health (those are actuator health indicators).

```bash
curl -i http://localhost:8080/webhook/health
# HTTP/1.1 200
# OK
```

## `GET /` — Dashboard

Read-only Thymeleaf dashboard listing recently analyzed PRs with their tier,
security flag, and one-click action links. No authentication; it only *reads*
`PrAnalysis` rows.

- Shows per-PR: repo, number, title, tier badge, security flag, when analyzed.
- Renders action buttons whose `href` is `/action?token=…&do=…` (see below).

Open in a browser: `http://localhost:8080/`.

## `GET /action` — One-click actions

Performs a maintainer action on a PR. **Requires a valid token.** The bot never
auto-merges; `close` only sets `state=closed`.

**Query params:**

| Param | Required | Meaning |
|---|---|---|
| `token` | yes | HMAC-signed, single-use token (15-min TTL) from the dashboard/email link. |
| `do` | yes | One of `approve`, `request-changes`, `close`. |

**Behavior:**

1. `TokenService.verify(token)` — rejects invalid/expired/already-used tokens
   with `401`/`403`.
2. `ActionController` performs the action via `GitHubApiClient` and marks the
   `PrAnalysis` row `ACTIONED` (+ `action_taken`).
3. Token is invalidated after use.

**Responses:**

| Status | Meaning |
|---|---|
| `200 OK` | Action performed; returns an HTML result page. |
| `400` | Invalid/expired/used token, or action mismatch. |
| `401` / `403` | Missing/invalid token. |
| `404` | PR not found. |
| `410` | Link already used / action already taken. |

> The token is generated by `TokenService` using `app.action-secret`; actions
> are only available when `app.action-secret` is set (see
> [how-to: setup → one-click actions](howto-setup.md#one-click-actions)).

## Actuator (observability)

Enabled endpoints: `health`, `info`, `prometheus`, `metrics`. Custom health
indicators exist for `github-api` and `llm`. Example:

```bash
curl http://localhost:8080/actuator/health
curl http://localhost:8080/actuator/prometheus
```

## Request correlation

Every log line carries a `traceId` equal to the webhook `X-GitHub-Delivery`
header, propagated across the async review threads. Grep a delivery id to follow
one PR end-to-end.
